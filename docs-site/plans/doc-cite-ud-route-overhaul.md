# Doc-Cite `u` Route Overhaul

## Goal

Reshape doc-cite generation so that:

- `docs-site/src/to-generate/docs/u/**` is the canonical universal concept tree
- `docs-site/src/to-generate/docs/lang/{lang}/**` is a strict subset projection
  of that tree
- each public `/{lang}/...` page is generated by rendering the matching
  `u/**/...` concept first, then rendering the matching `lang/{lang}/...`
  concept below it
- navigation for `/{lang}/...` is generated from what actually exists in that
  language tree
- public routes become UD-like slash routes such as `/u/pos/` and `/de/pos/`
  rather than the current `.html` routes

This is a clean break. The current doc-cite stubs exist only to explore the
shape and are killable. The only existing source with meaningful content that
must be treated carefully is
`docs-site/src/to-generate/docs/lang/de/rules/numerals.doc.ts`.

That content should move under `/{lang}/classification-instructions/` in the
new route model.

## Agreed Product Decisions

### Public route families

We keep both public route families:

- `dumling.xxx/u/**`
- `dumling.xxx/{lang}/**`

This intentionally mirrors UD:

- `https://universaldependencies.org/u/pos/`
- `https://universaldependencies.org/en/pos/`

### Universal tree is authoritative

`lang/{lang}` cannot invent concept pages that do not exist in `u/**`.

The language tree is a subset of the universal tree, not a peer tree.

### Omission granularity

For this overhaul, omission is page-level only.

Section omission can be added later once pages are generated from structured TS
objects and we have a stable way to address sections.

### Merge semantics

For a matching concept path:

1. render `u/.../concept.doc.ts`
2. render `lang/{lang}/.../concept.doc.ts` below it

This is append-only composition for now, not arbitrary field-level patching.

There should be no visible boundary by default. The language content should
continue the page naturally.

### Navigation source

Overview pages should live under `src/to-generate/docs/**` too, and navigation
for a language should be generated from the actual exposed language tree rather
than from a manually maintained overview list.

The overview intro prose still belongs in authored `*.doc.ts` files. To avoid
drift, those pages should lean on typed constrained data and generated
references rather than hand-written inventories.

### Route cleanup

Route cleanup is explicitly in scope.

The existing `.html` doc-cite route contract is disposable and should be
replaced with normal slash routes.

## Current State

The current generator already mirrors typed-doc source paths into public routes,
but the content ownership is inverted.

Today:

- `u/**` mostly contains stubs
- `lang/{lang}/**` mostly contains the actual user-facing bodies
- each source file is treated as one complete page
- there is no inheritance or overlay model
- doc-cite pages opt into a separate `htmlRoute` path contract

Relevant implementation seams:

- `docs-site/scripts/generate-content/docs/typed/load-typed-doc-source.ts`
- `docs-site/scripts/generate-content/docs/routes.ts`
- `docs-site/src/to-generate/docs/document-shapes.ts`
- `docs-site/src/lib/navigation.ts`
- `docs-site/src/pages/*.astro`

## Problems In The Current Model

### 1. Duplicated tree ownership

The `u` tree and the language tree currently have near-1:1 mirrored files:

- `u`: 88 files
- `lang/de`: 89 files

That is the wrong ownership model if `u` is supposed to define the concepts and
`lang/{lang}` is supposed to select from them.

### 2. Wrong content authority

The language file currently owns the whole public page body.

That means universal concept prose cannot be written once and reused across
languages.

### 3. Brittle route special-casing

The current doc-cite system carries both:

- `routeId`
- `htmlRoute`

and uses dedicated Astro route files for `.html` depth variants:

- `[scope].html.astro`
- `[scope]/[page].html.astro`
- `[scope]/[section]/[leaf].html.astro`
- `[scope]/[section]/[subsection]/[leaf].html.astro`

That entire route layer exists only because doc-cite pages were forced into a
special `.html` contract. With UD-like slash routes, this should collapse back
to the normal catch-all docs routing model.

### 4. Overview drift risk

If overview pages are authored as long manual inventories, they will drift from
the actual exposed subset.

The language tree itself should drive navigation and overview listings.

## Target User-Facing Outcome

For a user browsing `/{lang}/...`:

- the tree should feel complete without needing `/u/...`
- every exposed page should explain both the universal concept and the
  language-specific reality
- missing concepts should simply not appear in that language tree
- parent overview pages should truthfully reflect the currently exposed children
- language-local classifier guidance should remain available under
  `/{lang}/classification-instructions/...`

For a user browsing `/u/...`:

- the tree should present the universal conceptual vocabulary
- it should not pretend to describe every language-specific realization

## Proposed Generation Model

### Source roles

There are two typed-doc source kinds.

`u/**`

- defines canonical concept identity
- defines universal prose and universal structure
- is publicly routable under `/u/...`

`lang/{lang}/**`

- declares exposure of a matching universal concept
- contributes language-specific prose, examples, and commentary
- is publicly routable under `/{lang}/...`

There is also one explicit language-only route family:

`lang/{lang}/classification-instructions/**`

- does not need a `u/**` counterpart
- is public under `/{lang}/classification-instructions/...`
- is for classifier guidance such as the existing German numerals notes
- is a permanent narrow exception, not a general escape hatch for arbitrary
  language-only sections

### Matching rule

A language concept source must map to an existing universal concept source at
the same relative concept path.

Examples:

- `u/feature/case.doc.ts` <-> `lang/de/feature/case.doc.ts`
- `u/pos/noun.doc.ts` <-> `lang/de/pos/noun.doc.ts`
- `u.doc.ts` <-> `lang/de.doc.ts`

If a language file exists without a universal counterpart, generation should
fail.

### Final page rendering

For `/u/...` routes:

- render the universal concept page only

For `/{lang}/...` routes:

- render the universal concept page
- render the language overlay content below it

The key point is that the language page is not a replacement page. It is a
composed page.

### Navigation generation

Navigation should be generated from emitted public pages, not from hard-coded
overview prose.

That means:

- `/u/...` nav reflects the universal tree
- `/{lang}/...` nav reflects only the pages that exist in that language subset

Overview pages can still contain prose, but their child listings should be
derived from the tree that actually exists.

That means overview docs should be authored around generated facts rather than
manual inventories. Good candidate inputs are:

- typed descendant registries
- typed inventory constants checked with `satisfies`
- generated attestation doc references
- generated CSV or descriptor IDs when concrete examples or anchors are needed

## Recommended Data Model Shift

The current `SourceMirroredDocCitePageDefinition` shape is too page-centric for
this new model.

We should split it into two explicit source definitions.

### Universal concept page

This should represent the canonical concept page that can stand on its own under
`/u/...`.

It likely needs:

- page meta
- family
- subject
- universal body
- universal examples
- universal subsections

Universal examples are allowed, but they should remain a distinct source-layer
concept instead of being silently merged into language-local example lists.

When generating `/{lang}/...` pages, universal examples should be omitted
entirely. Each language page is responsible for providing its own examples.

### Language concept overlay

This should represent additive language-local content for the matching concept.

It likely needs:

- page meta overrides only where truly needed
- language-local body
- language-local examples
- language-local subsections

It should not define independent route identity.

The route comes from the matched universal path plus the `{lang}` scope.

When generating a `/{lang}/...` page, universal and language examples should
stay distinguishable in the source model even if the final page renders them in
one continuous flow.

## Recommended Route Model Shift

We should remove doc-cite dependence on explicit `htmlRoute` frontmatter and let
doc pages use the same slash-route conventions as the rest of the docs site.

Target examples:

- `/u/`
- `/u/feature/`
- `/u/feature/case/`
- `/de/`
- `/de/feature/`
- `/de/feature/case/`

This likely means:

- stop requiring `doc.htmlRoute`
- stop storing special `.html` values in generated frontmatter for doc-cite
  pages
- stop using the depth-specific `.html.astro` route files for doc-cite pages
- route everything through the normal catch-all doc page flow

## Proposed Implementation Steps

### Phase 1. Replace route contract

1. remove doc-cite dependence on `.html` routes
2. normalize doc-cite public links to slash routes
3. simplify Astro page routing so doc-cite pages use the same routing model as
   other docs pages

This should happen early because it simplifies every later step.

### Phase 2. Split source definitions

1. introduce a universal concept page definition helper
2. introduce a language overlay definition helper
3. update typed-doc loading so it can distinguish universal pages from language
   overlays

### Phase 3. Add universal-to-language composition

1. discover all `u/**` concept pages
2. discover all `lang/{lang}/**` overlay pages
3. validate that every language overlay has a universal counterpart
4. emit `/u/...` pages from universal sources
5. emit `/{lang}/...` pages by composing universal + language sources
6. emit `lang/{lang}/classification-instructions/**` as a parallel
   language-only typed-doc section

For `classification-instructions/` specifically:

- `u/classification-instructions/index.doc.ts` provides the universal index
  content
- `lang/{lang}/classification-instructions/index.doc.ts` renders that universal
  index content plus a generated list of the language-local `how-to-*` pages
- `lang/{lang}/classification-instructions/how-to-*.doc.ts` pages are the
  allowed language-only leaves in this section

### Phase 4. Drive navigation from emitted trees

1. generate nav from the actual emitted page set
2. make overview child listings derive from present descendants
3. stop hand-maintaining long overview inventories where the tree itself already
   knows the answer

### Phase 5. Migrate content ownership

1. move concept-defining prose out of `lang/{lang}` into `u/**`
2. keep language-local examples and commentary in `lang/{lang}`
3. preserve and relocate the meaningful German `numerals` content into the new
   `de/classification-instructions/` route family
4. delete the current throwaway stubs

## Validation Rules

Generation should fail when:

- a `lang/{lang}` concept file has no matching `u/**` concept file
- two sources resolve to the same public route
- a generated overview links to a child page that is not actually emitted
- a mirrored language concept page tries to live outside the allowed subset tree
- a language-only doc exists outside `lang/{lang}/classification-instructions/`
- a `classification-instructions` language-only leaf does not follow the
  `how-to-*` naming pattern

Generation should guarantee:

- `/u/...` is the full universal concept tree
- `/{lang}/...` is a subset of `/u/...`
- each `/{lang}/...` page shows universal content first and language content
  second
- `/{lang}/classification-instructions/...` remains available as an intentional
  language-only exception
- `/{lang}/...` pages do not inherit universal examples
- `/{lang}/classification-instructions/` index combines the universal index
  content with a generated list of language-local how-to pages

## Pressure-Test Questions Before Coding

The broad shape is settled. These are the remaining questions that still decide
whether the generator stays coherent.

### 1. How hard should we make it to hand-type overview facts incorrectly?

You want authored overview intros in `*.doc.ts`, but backed by typed constraints
and generated references.

The implementation question is how strict to be:

- require child lists to come from generated descendant registries
- require concrete example links to reference generated attestation docs
- expose helper APIs that build inventories from typed constants checked with
  `satisfies`

My bias is to make the common drift-prone lists impossible to hand-maintain
wrongly.

### 2. Do we want dedicated helpers for generated overview lists now or later?

The `classification-instructions` index and the tree-backed overview pages both
want generated child lists.

The open implementation choice is whether to:

- build that with a small generic descendant-list helper now
- hand-roll the first version in the doc generator, then extract helpers later

Given your "not strict for now" answer, my bias is to keep the first iteration
simple and only extract helpers once the second use site appears.
